1 · Executive Summary

UP Labs คือแพลตฟอร์มสุขภาพ-longevity ของ UP Wellness (Next.js 14 + Supabase) ที่รวมเครื่องมือ 26 ตัว: จัดการลูกค้า (Customer 360), วัดองค์ประกอบร่างกาย (BCA), wearable intelligence (UP Pulse), คัดกรอง prospect (CheckForm + STP), รายงาน (Med-Map / Longevity Lab Report), โภชนาการ (NutriScan / Plate Planner), LINE Bot, Admin/RBAC ฯลฯ

ปัญหาที่ v2 แก้: v1 ถูกติว่า "ไม่สวย ใช้ยาก" — รากของปัญหาคือ UI ไม่ตรงกับ brand ของตัวเอง (Sage + Everyman + Caregiver / "หมอที่เป็นเพื่อน") เพราะใช้ glassmorphism + aurora เคลื่อนไหวหนัก, emoji เป็นไอคอน, label ตัวพิมพ์ใหญ่ monospace, ตัวอักษรเล็ก (9–13px), ระบบสีสถานะซ้อน 3 ชุด, ไม่มี nav ข้ามแอป, และลูกค้า 1 คนมี 2 หน้า (/customers vs /pulse/master)

2 · Goals & Non-Goals

Goals (P0)

  • คงฟีเจอร์ครบ 100% — ทุกหน้า/ปุ่ม/ฟังก์ชันของ v1 ต้องมีใน v2 (ตรวจด้วย parity matrix ทุก module)
  • โครงสร้าง/เส้นทางข้อมูลเดิม — ใช้ Supabase schema เดิม + API routes เดิม (เปลี่ยนแค่ presentation layer)
  • Redesign UI ให้ใช้ง่าย — clinical-warm: สะอาด คม อ่านชัด อุ่น มี nav/breadcrumb ชัด ลดของตกแต่งที่กลบเนื้อหา
  • ★ Customer Profile + BCA ต้องโชว์ identity block: ชื่อ · วันเกิด (ค.ศ.) · อายุ · เพศ · ส่วนสูง (ดู §4)
  • เข้าถึงง่าย/มือถือดี — responsive, touch ≥44px, loading/empty/error ครบทุก state

Non-Goals (ไม่ทำในรอบนี้)

  • ไม่เปลี่ยน schema / ไม่ migrate data / ไม่แตะ RLS (รวม co-coach assignment ที่เพิ่งทำ)
  • ไม่ลบฟีเจอร์ใด ไม่เปลี่ยน business logic (การคำนวณ score/BCA/PV ฯลฯ เหมือนเดิม)
  • ไม่ทำฟีเจอร์ใหม่ที่ไม่มีใน v1 (ยกเว้น identity block + การจัดวาง/nav ที่ช่วยให้ใช้ง่าย)
  • ไม่เปลี่ยน auth provider / ไม่เพิ่ม dependency หนักโดยไม่จำเป็น

3 · กลยุทธ์ v2 — สร้างขนาน (parallel), สลับเมื่อพร้อม

v2 ถูกสร้างใต้เส้นทาง /v2/* โดย ใช้ API + ข้อมูลจริงชุดเดียวกับ v1 (ไม่ใช่ mock) เพื่อให้เปรียบเทียบ v1↔v2 ได้ตรงๆ และ v1 ยังทำงานได้ตลอดระหว่างพัฒนา

เฟสv1 (เดิม)v2 (ใหม่)
ระหว่าง devใช้งานจริงปกติ/v2/* เปิดทดสอบคู่ขนาน (real data)
parity ผ่าน/ moduleยังอยู่BA+User เซ็นรับ
cutoverredirect → /v2 หรือ feature flagกลายเป็น default
หลัง cutoverเก็บเป็น ?legacy=1 ชั่วคราวproduction
หลักการ reuse: v2 เรียก /api/customers/[id]/360, /api/customers/list, /api/customers/[id]/measurements ฯลฯ ตามเดิม — เปลี่ยนเฉพาะ component/layout. ถ้าพบว่า API ขาดข้อมูลที่ identity block ต้องใช้ (เช่น birth_date, height) ให้ เพิ่ม field ใน response เท่านั้น (ไม่ทำลายของเดิม)

4 · ★ Customer Identity Block (ข้อกำหนดบังคับ)

บังคับ: ทั้ง Customer Profile (360) และหน้า BCA ต้องแสดงข้อมูลลูกค้าเหล่านี้เด่นชัดเสมอ

Fieldแหล่งข้อมูลรูปแบบแสดงผล
ชื่อcustomers.name (+ nickname ถ้ามี)"ชื่อเต็ม (ชื่อเล่น)"
วันเดือนปีเกิดcustomers.birth_dateค.ศ. (Gregorian) เช่น 7 Nov 1982 หรือ 1982-11-07 — ไม่ใช้ พ.ศ.
อายุคำนวณจาก birth_date ถึงวันนี้ (fallback: birth_year)"43 ปี" (เลขจำนวนเต็มปี)
เพศcustomers.gender (male/female/other)"ชาย / หญิง / อื่นๆ" + ไอคอน ♂ ♀
ส่วนสูงcustomers.height (cm)"165 ซม." (ใช้คำนวณ BMI ในหน้า BCA)

กฎการแสดงผล

  • อายุ คำนวณจาก birth_date เป็นหลัก (แม่นระดับวัน) — ถ้าไม่มี birth_date ใช้ birth_year (ปีปัจจุบัน − birth_year)
  • วันเกิดแสดงเป็น ค.ศ. เสมอ (ผู้ใช้ระบุชัด) — ถ้าจะโชว์ พ.ศ. ควบ ให้เป็น secondary เล็กๆ
  • ถ้า field ใดว่าง → แสดง "—" และมีปุ่ม/ลิงก์ "เพิ่มข้อมูล" (แก้ที่หน้า edit customer เดิม)
  • หน้า BCA: identity block อยู่ส่วนหัว เหนือฟอร์ม/ตารางวัด — ผู้ใช้เห็นว่ากำลังบันทึกให้ใคร + ใช้ height คำนวณ BMI
ต้องตรวจว่า /api/customers/[id]/360 และ data ของหน้า BCA ส่ง name, nickname, birth_date, birth_year, gender, height มาครบ. ปัจจุบัน 360 ส่ง customer object เต็ม (select("*")) — มีครบแล้ว; ยืนยันว่า IdentityBar v2 ใช้ field เหล่านี้จริง.

5 · Design System v2 — "Clinical-Warm"

คง token สีเดิม (rose/wellness/science/amber/ink/status) แต่เปลี่ยนวิธีใช้: เลิก glass หนา + aurora เคลื่อนไหว → พื้นนิ่ง การ์ดขาวคม เงานุ่ม

สี (คงเดิม — ใช้เป็นระบบเดียว)

Rose #8C4C4C
primary · brand warmth · ปุ่มหลัก/ลิงก์
Wellness #396755
positive · สุขภาพดี · co-coach
Science #2A7B8F
data · กราฟ · ลิงก์อ้างอิง

Status — ระบบเดียว (เลิก 3 ชุดซ้อน)

optimal good caution warning danger

map ครั้งเดียวใน lib/status → ใช้ทั้ง badge/กราฟ/ข้อความ. กราฟห้าม hex ลอย ต้องอ้าง token.

Typography

  • หัวข้อ/ตัวเลข: Manrope (head) · เนื้อหาไทย: Sarabun (thai) · โค้ด/timestamp: JetBrains Mono
  • เลิก label monospace UPPERCASE tracking กว้างทั่วทุกที่ → ใช้ sentence-case text-[12px] font-semibold text-ink-60 (mono เก็บไว้เฉพาะตัวเลข/โค้ด)
  • ขนาดตัวอักษรเล็กสุดขยับขึ้น 9–10px → ≥12px; สเกล 6 ระดับ (12/14/15/17/23/34)

Components (มาตรฐานเดียว)

Componentกติกา v2
Cardพื้นขาว rounded-2xl border border-ink-10 + เงานุ่ม (ไม่ใช้ .liquid)
Button3 ระดับชัด: primary (rose solid) · secondary (outline) · ghost. ปุ่ม async โชว์ spinner เสมอ
IconLucide 16–20px strokeWidth 2 (เลิก emoji-as-icon; emoji เฉพาะที่สื่อความหมายจริง)
Status pillrounded-full + dot + label จากระบบเดียว
Tabsเส้นใต้/segmented เรียบ ไม่ใช่ glass shell; active ชัด; keyboard นำทางได้
Tableหัวตารางนิ่ง, row hover, มือถือ scroll มีเงาบอกขอบ + สลับเป็น card list
Form fieldlabel ชัด, error inline, touch ≥44px, save state ชัด
Empty/Loading/Errorครบทุก tab/หน้า โทนสม่ำเสมอ (Lucide + ข้อความไทยอุ่นๆ + action)

Background & Motion

  • พื้น warm-white นิ่ง (เลิก aurora 3 orbs animate) — gradient คงที่บางๆ ได้เฉพาะ hero
  • เคารพ prefers-reduced-motion / prefers-reduced-transparency / prefers-contrast + print = solid

6 · Global Patterns (แก้ปัญหา "หาทางไม่เจอ")

  • App Shell + Nav ถาวร: top bar มีโลโก้ + app switcher (เข้าถึง 26 apps จากทุกหน้า) + ค้นหา + เมนูผู้ใช้ — ไม่ต้องกลับ hub ทุกครั้ง
  • Breadcrumb ทุกหน้า detail: เช่น ลูกค้า › นส.เอ › BCA
  • ยุบหน้าลูกค้าซ้ำ: /pulse/master ผนวกเป็น "tab Pulse" ใน Customer 360 (ลิงก์ถึงกัน; ลูกค้า 1 คน = 1 หน้าเข้าใจง่าย)
  • Page header มาตรฐาน: ชื่อหน้า + คำอธิบายสั้น + primary action ขวาบน
  • Search/filter วางตำแหน่งเดิมทุกหน้า list, จำ state
  • ฟอร์มยาวมี progress (เช่น CheckForm หลาย section)
  • a11y: aria-label ปุ่มไอคอน, สถานะมี label ไม่พึ่งสีล้วน, focus-visible ชัด, contrast ≥4.5:1
  • i18n: ไทยเป็นหลัก (Sarabun), ศัพท์เทคนิคไทย-อังกฤษควบ, รองรับผู้ใช้พิมพ์อังกฤษ
